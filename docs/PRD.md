## 零、技术架构总览（新增）

### 0.1 架构原则

| 原则          | 说明                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------- |
| **前后端分离**   | 前端（Taro + Vue3/TS）与后端（FastAPI + Python）完全解耦，仅通过 RESTful API 通信                              |
| **个人测试号开发** | 当前使用微信小程序个人测试号 AppID，所有功能不依赖正式号专属能力（如微信支付、模板消息、手机号授权）                                       |
| **一键迁移正式号** | 迁移仅需：(1) 更换 `project.config.json` 中的 AppID；(2) 在微信管理后台配置服务器域名白名单；(3) 更新前端环境变量中的 API BaseURL |
| **域名可配置**   | 前端 API 基地址通过 `config.ts` 集中管理，支持 dev/staging/prod 三套环境切换，不硬编码                               |

### 0.2 技术栈

```
┌─────────────────────────────────────────────────────┐
│          前端（Taro 3.x 微信小程序）                   │
│  TypeScript + Vue3 + Pinia + Taro UI + ECharts      │
│  wx.login() → openid → 后端 JWT 鉴权                 │
│  语音输入：wx.getRecorderManager / wx.startRecord    │
│  文件上传：wx.chooseMessageFile                     │
└──────────────────────┬──────────────────────────────┘
                       │ HTTPS / RESTful API
┌──────────────────────▼──────────────────────────────┐
│          后端（FastAPI + Python 3.11+）               │
│  SQLAlchemy ORM + SQLite + JWT + bcrypt              │
│  DeepSeek API（AI 规划 & 推荐）                       │
│  APScheduler / Celery（定时任务：遗忘曲线召回）         │
└─────────────────────────────────────────────────────┘
```

### 0.3 环境配置对照

| 配置项              | 开发环境（测试号）                       | 生产环境（正式号）                              |
| ---------------- | ------------------------------- | -------------------------------------- |
| AppID            | 个人测试号 AppID                     | 正式号 AppID                              |
| API BaseURL      | `http://localhost:8000` 或内网穿透域名 | `https://api.example.com`              |
| 服务器域名白名单         | 测试号不校验                          | 微信管理后台配置 `request` / `uploadFile` 合法域名 |
| 数据库              | SQLite（本地文件）                    | SQLite（后续可迁移至 PostgreSQL）              |
| DeepSeek API Key | 开发 Key（环境变量）                    | 生产 Key（环境变量）                           |

### 0.4 微信小程序鉴权流程

```
用户打开小程序
    ↓
wx.login() 获取临时 code
    ↓
前端 POST /auth/wechat-login { code }
    ↓
后端调用微信 code2Session 换取 openid
    ↓
后端查找/创建用户 → 生成 JWT token → 返回 { access_token, user }
    ↓
前端存储 token，后续请求统一携带 Authorization: Bearer <token>
```

> 当前以 `wx.login()` + openid 为主鉴权方式。邮箱密码登录作为开发调试备选，不入小程序前端入口。

### 0.5 个人测试号限制与应对

| 测试号限制      | 影响                                      | 应对方案                                                |
| ---------- | --------------------------------------- | --------------------------------------------------- |
| 无模板消息      | 无法使用 `wx.requestSubscribeMessage` 做服务通知 | 打卡提醒使用小程序内 `wx.showModal` 本地弹窗 + 定时轮询；代码预留正式号模板消息接口 |
| 无手机号授权     | 无法获取微信绑定手机号                             | 不依赖手机号，仅通过 openid 标识用户                              |
| 请求域名不校验    | 开发阶段可直接调用本地服务                           | 代码中不写死域名，全部走 `config.ts`                            |
| 最多 15 个体验者 | 无法大规模分发测试                               | 满足内部开发测试，正式号发布后无此限制                                 |
| 无法发布上线     | 仅用于开发调试                                 | 代码不依赖测试号特有 API，正式号一键迁移                              |
| 无微信支付      | 不涉及                                     | 产品无支付功能                                             |

### 0.6 权限模型

| 层级 | 判断方式 | MVP 范围 |
| --- | --- | --- |
| **普通用户** | 默认（users.role='user'） | 全部打卡/统计/规划/排行功能 |
| **小组管理员** | groups.creator_id == user.id | 更新自己创建的小组信息；管理功能（移除成员等）后续迭代 |
| **系统管理员** | users.role='admin'（DB 手动设置） | 预留 require_admin() 中间件，后续开发 /admin/* 端点 |

> MVP 阶段所有用户 role 均为 `'user'`，管理员功能预留接口但不实现。

## 一、项目背景与目标

### 1.1 项目背景

在当今快节奏的学习环境中，学生和自律学习者需要一种简单有效的方式来记录和追踪学习进度。传统的打卡方式（如纸质打卡、Excel记录）缺乏可视化反馈和社交激励，难以长期坚持。同时，普通打卡工具只关注"记录行为"和"连续天数"，无法真正辅助学习和复习，也无法引导用户建立规律的学习习惯。

本项目旨在打造一款**智能学习打卡工具**，通过 AI 辅助规划、遗忘曲线复习机制、智能日历展望和规律性排行，将"打卡"与"学习"真正打通，帮助用户建立规律的学习习惯并有效巩固知识。

### 1.2 项目目标

| 目标类型 | 描述                                               |
| ---- | ------------------------------------------------ |
| 核心目标 | 打造微信小程序学习打卡工具，支持每日打卡、AI 辅助规划、遗忘曲线复习、智能日历展望、规律性排行 |
| 用户目标 | 通过每日打卡、数据统计、AI 辅助和社交激励，提升学习动力和成就感，建立稳定的学习习惯      |
| 业务目标 | 构建用户粘性强、具有传播性的习惯养成产品，成为学习辅助领域的差异化产品              |

### 1.3 产品定位

| 维度    | 描述                                                                     |
| ----- | ---------------------------------------------------------------------- |
| 产品名称  | 学习打卡追踪器                                                                |
| 产品形态  | 微信小程序                                                                  |
| 目标用户  | 学生群体、自律学习者、备考人群、习惯养成爱好者                                                |
| 核心价值  | 每日打卡 → 连续记录 → AI 智能规划 → 未来展望日历 → 规律性排行 → 成就感激励                         |
| 差异化亮点 | 从"记录行为"升级为"辅助学习"，从"手动规划"升级为"AI 辅助规划"，从"回顾过去"升级为"展望未来"，从"比打卡久"升级为"比更规律" |

## 四、功能列表

### 4.1 页面结构（与原型对照）

根据高保真原型，小程序包含 6 个底部 Tab 页面：

| Tab | 页面                | 核心功能组件                                                                                                           |
| --- | ----------------- | ---------------------------------------------------------------------------------------------------------------- |
| 首页  | `/pages/home`     | 连续天数 Hero 区、三指标卡片（本周/本月打卡率、规律性排名）、今日任务列表（一键勾选打卡）、任务状态标签（已同步/主任务/轻量）                                              |

## 五、业务流程

### 5.1 打卡流程

```
用户进入首页 → 查看今日任务列表 → 点击未打卡任务的复选框
    → 后端记录打卡时间（支持每日多次）
    → 更新连续天数和规律性得分
    → 检查是否触发遗忘曲线复习任务生成
    → 检查是否触发明日任务推荐
    → 前端即时更新界面状态
```

### 5.2 AI 任务生成流程

```
用户进入规划页 → 选择学习模式（或留空，在提示词中描述，均无则默认"自由模式"）
    → 输入/粘贴学习材料（可含目标时限等提示词）或语音/文件上传
    → 点击"AI 规划"按钮
    → 前端 POST /ai/generate-plan { content, mode, target_days? }
    → 后端提取模式：优先用 mode 参数，为空则从 content 中 NLP 推断（如"仅剩30天"→考试冲刺），均无法推断→默认"free"
    → 后端按对应模式策略组装 Prompt → 调用 DeepSeek API
    → AI 返回结构化 JSON：每个子任务含 task_name + scheduled_date（具体日期）+ subject + suggested_duration + difficulty
    → 前端渲染可编辑日期计划表（按日期排序，每个日期展开显示当天子任务）
    → 用户可调整：子任务名称、执行日期、删除子任务
    → 用户也可手动通过"自定义计划"区域额外添加任务
    → 点击"确认创建计划" → 前端 POST /tasks/batch（每个子任务携带自己的 start_date）
    → 跳转首页，日历页同步显示新安排的日期任务
```

### 5.3 智能日历展望流程

```
用户进入日历页 → 系统定位今天格子（深色）→ 显示图例说明
    → 下方展示"今日任务"卡片（当天安排的任务）
    → 下方展示"明日推荐"卡片（AI 根据当前进度动态生成）
    → 点击任意日期弹出 Sheet → 显示当天详情
    → 未来日期：显示已安排学习任务；过去日期：显示打卡记录
    → 连续打卡 ≥7 天时，推荐休息日（灰色小点标记）
```

### 5.4 规律性排行计算流程

```
用户完成打卡 → 后端记录打卡时间
    → 计算用户所有打卡时间的标准差和波动系数
    → 规律性得分 = 100 - (标准差/平均打卡时间) × 100
    → 与组内/全局用户对比 → 确定排名区间（前20%/20-60%/60%+）
    → 检查徽章授予条件 → 自动授予达标徽章
    → 前端获取排名和徽章数据 → 渲染排行页
```

## 十、边界条件与异常处理

### 10.1 打卡相关

| 场景        | 处理方式                     |
| --------- | ------------------------ |
| 当天未打卡     | 日历显示未打卡，统计计入未打卡天数        |
| 补打卡超过 7 天 | 后端拒绝并返回错误提示              |
| 连续打卡中断    | 连续天数重置为 0                |
| 同一任务多次打卡  | 允许多次打卡                   |
| 任务过期      | 自动标记为 expired，不再显示在待打卡列表 |

### 10.2 AI 辅助相关

| 场景           | 处理方式                        |
| ------------ | --------------------------- |
| AI 解析失败      | 显示错误提示，引导用户重新输入或简化内容        |
| AI 返回格式异常    | 前端做数据校验，格式错误时提示重试           |
| AI 调用超时      | 显示加载超时提示，提供重试按钮             |
| 超过每日调用次数     | 返回 429 错误 + "今日 AI 规划次数已用完" |
| 用户拒绝 AI 生成任务 | 任务不创建，用户可重新编辑或手动创建          |

### 10.3 日历相关

| 场景       | 处理方式                    |
| -------- | ----------------------- |
| 用户无今日任务  | 今日任务卡片显示"暂无任务"，引导去规划页创建 |
| 今日任务全部完成 | 明日推荐卡片自动更新              |
| 用户拒绝休息建议 | 可继续打卡，休息标记保留但不强制        |

### 10.4 排行相关

| 场景         | 处理方式                        |
| ---------- | --------------------------- |
| 打卡次数 < 7 次 | 排行页显示"数据不足，继续打卡"            |
| 跨时区打卡      | 按用户设置时区（默认 Asia/Shanghai）计算 |

### 10.5 通用异常

| 场景    | 处理方式                        |
| ----- | --------------------------- |
| 网络断开  | 前端显示离线提示；打卡操作本地暂存，恢复后同步（可选） |
| 重复提交  | 前端防抖处理，防止同一任务重复打卡           |
| 首次使用  | 首页显示空状态引导，引导用户去规划页创建首个任务    |
| 无打卡记录 | 统计页显示"暂无数据"                 |

| 步骤 | 操作               | 说明                                                                   |

| 5  | 启用模板消息           | 在代码中启用已预留的 `wx.requestSubscribeMessage` 逻辑，替换测试号的本地弹窗提醒              |
