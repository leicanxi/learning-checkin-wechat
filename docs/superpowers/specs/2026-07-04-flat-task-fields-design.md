# Flat Task Fields Design

Date: 2026-07-04

## Goal

Unify the learning check-in business model around flat daily task instances. Every task row represents one concrete task on one concrete date. User-created schedules, AI plans, and future review logic are only ways to generate task rows; they are not runtime rules used by home, calendar, check-in, or statistics pages.

This design intentionally simplifies the MVP. It removes repeated interpretation of `repeat_days`, `start_date`, `end_date`, `Task.status=completed`, review flags, and frontend-only completion logic from core runtime flows.

## Core Principle

Runtime business pages only read task instances:

```text
Task on a date + optional checkin = UI state
```

Examples:

```text
背单词 / 2026-07-03 / 英语 / 30min
背单词 / 2026-07-04 / 英语 / 30min
```

If the same learning action appears on multiple dates, it is stored as multiple task rows with the same name and different `task_date` values.

## Data Model

### tasks

The existing `tasks` table should be treated as a task-instance table in MVP.

Canonical fields:

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | integer | Task instance id |
| `user_id` | integer | Owner |
| `name` | string | Task name, such as `背单词` |
| `task_date` | date | The date this task should be done |
| `subject` | string | User-defined subject text |
| `suggested_duration` | integer | Suggested minutes |
| `difficulty` | string | `low`, `medium`, or `high` |
| `knowledge_tags` | JSON text | Knowledge tags generated by AI or entered by user |
| `source` | string | `manual`, `ai`, or `system` |
| `status` | string | Lifecycle only: `active`, `deleted`, or `expired` |
| `created_at` | datetime | Creation time |
| `updated_at` | datetime | Last update time |

`status` must not represent whether the task has been checked in. Completion is derived from check-ins.

### checkins

Canonical fields:

| Field | Type | Meaning |
| --- | --- | --- |
| `id` | integer | Check-in id |
| `user_id` | integer | Owner |
| `task_id` | integer | Linked task instance |
| `checkin_date` | date | User-local check-in date |
| `checkin_time` | datetime | Actual check-in timestamp |
| `subject` | string | Subject snapshot at check-in time |
| `created_at` | datetime | Creation time |

Uniqueness rule:

```text
user_id + task_id + checkin_date
```

One task instance can be checked in at most once per local date.

## Frozen Legacy Fields

These fields may remain in the database temporarily for compatibility, but MVP runtime logic must not depend on them:

```text
repeat_days
start_date
end_date
task_type
is_review_task
parent_task_id
ai_review_intervals
review_round
Task.status = completed
Checkin.is_review
```

If keeping old columns temporarily, backend serializers should map old storage to the new canonical API shape. New frontend code should not read or write the legacy names.

## API Contract

### Create Task

All creators use the same canonical request shape.

```json
{
  "name": "背单词",
  "task_date": "2026-07-03",
  "subject": "英语",
  "suggested_duration": 30,
  "difficulty": "medium",
  "knowledge_tags": ["四级词汇"],
  "source": "manual"
}
```

### Batch Create Tasks

Manual repeated selections and AI plans both become a list of task instances.

```json
{
  "tasks": [
    {
      "name": "背单词",
      "task_date": "2026-07-03",
      "subject": "英语",
      "suggested_duration": 30,
      "difficulty": "medium",
      "knowledge_tags": ["四级词汇"],
      "source": "manual"
    }
  ]
}
```

### List Tasks

The list endpoint should support date and range filtering:

```text
GET /tasks?date=2026-07-03
GET /tasks?start=2026-07-01&end=2026-07-07
```

Response items include derived check-in state:

```json
{
  "id": 1,
  "name": "背单词",
  "task_date": "2026-07-03",
  "subject": "英语",
  "suggested_duration": 30,
  "difficulty": "medium",
  "knowledge_tags": ["四级词汇"],
  "source": "manual",
  "status": "active",
  "completed": true,
  "checkin_id": 12
}
```

`completed` and `checkin_id` are derived response fields. They are not stored on `tasks`.

### AI Generate Plan

The frontend-facing AI response should use the same task fields as normal task creation:

```json
{
  "summary": "7 天背单词计划",
  "tasks": [
    {
      "name": "背单词 Unit 1",
      "task_date": "2026-07-03",
      "subject": "英语",
      "suggested_duration": 30,
      "difficulty": "medium",
      "knowledge_tags": ["Unit 1"],
      "source": "ai"
    }
  ]
}
```

If the AI provider internally returns `task_name` or `scheduled_date`, conversion must happen inside the backend AI router. The frontend should not consume AI-specific field names.

### Check In

```json
{
  "task_id": 1
}
```

The backend sets `checkin_date` from the user's local business date and `checkin_time` from the actual timestamp.

## Frontend Mapping

### Home

Home should call:

```text
GET /tasks?date=today
```

It renders the returned tasks directly. The completion checkbox uses `completed`. Cancel check-in uses `checkin_id`.

Home must stop combining `/tasks` and `/checkins` to derive its own completion state.

### Calendar

Calendar month should be based on task instances and check-ins in the month.

Date status:

| Condition | Status |
| --- | --- |
| Date has tasks and all are completed | `checked_in` |
| Date has tasks and at least one is incomplete, date is today or future | `pending` |
| Date has tasks and at least one is incomplete, date is past | `missed` |
| Date has no tasks | `empty` |

Rest and review statuses are not part of the MVP runtime model.

### Planner

Planner creates task instances.

Manual repeated selections are expanded before or during batch creation. The runtime model never stores repeat rules as active task logic.

AI plans already contain dates, so they become task instances directly.

### Stats

The main completion-rate definition is:

```text
completed task count / scheduled active task count
```

For a date range:

```text
scheduled active task count = tasks where task_date is in range and status = active
completed task count = scheduled active tasks with a checkin
```

MVP subject distribution uses completed tasks only:

```text
subject distribution = completed checkins grouped by checkins.subject
```

If a later page needs planned-task distribution, it should be added as a separate metric with a separate name.

### Settings and Groups

Group completion rate should use the same task-instance completion-rate definition as stats.

## Naming Rules

Canonical names:

| Concept | Canonical Field |
| --- | --- |
| Task name | `name` |
| Task date | `task_date` |
| Completion state | `completed` |
| Check-in date | `checkin_date` |
| Actual check-in time | `checkin_time` |
| Source | `source` |
| Knowledge tags | `knowledge_tags` |
| Subject | `subject` |
| Suggested duration | `suggested_duration` |

Avoid these names in new frontend or API code:

```text
task_name
scheduled_date
done
isDone
completedStatus
start_date
end_date
```

`done` may only exist as a local UI variable during migration, not as an API contract.

## Migration Strategy

The safest implementation path is:

1. Add canonical fields while temporarily keeping legacy columns.
2. Update Pydantic schemas to expose the canonical API contract.
3. Update task creation and batch creation to accept `task_date`, `knowledge_tags`, and `source`.
4. Update check-in creation to stop mutating `Task.status`.
5. Update task listing to return `completed` and `checkin_id`.
6. Update home, calendar, planner, stats, and settings to consume the canonical fields.
7. Freeze review fields in UI and runtime logic for MVP.
8. After data is stable, remove or migrate legacy fields in a separate database migration.

## Testing Requirements

Backend tests should cover:

- Creating a manual task instance.
- Batch creating AI task instances.
- Two same-name tasks on different dates are independent.
- One task can only be checked in once per local date.
- Check-in does not change `Task.status`.
- Listing tasks by date returns `completed` and `checkin_id`.
- Completion rate equals completed task instances divided by scheduled active task instances.

Frontend smoke checks should cover:

- Home displays only today's task instances.
- Checking in toggles `completed` through backend response state.
- Calendar date status reflects task instances and check-ins.
- Planner sends canonical `name` and `task_date`, not `task_name` and `scheduled_date`.

## Out of Scope for MVP

- Automatic review task generation.
- Long-running task-series renewal.
- User-managed subject dictionary table.
- Infinite or rolling generation of repeated tasks.
- Multi-check-in records for the same task on the same date.
