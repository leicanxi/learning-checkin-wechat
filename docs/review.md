### 3.2 缺失场景（MISS — 5 项）

| # | 缺失场景 | 严重度 | 建议处理 |
|---|---------|:---:|---------|
| MISS-1 | **JWT token 过期** — 用户在 APP 内长时间停留后 token 失效，再次调用 API 返回 401，但体验层未定义处理方式 | **高** | 前端拦截器检测 401 → 清除 token → 跳转登录页 → 静默重授权（wx.login 重新获取 token）→ 恢复用户操作（不丢失上下文） |
| MISS-2 | **sqlite 数据库文件损坏 / 磁盘写满** — 单文件数据库在异常断电或磁盘满时可能损坏 | **高** | 1) 生产环境增加 WAL 模式 + 定期 VACUUM；2) 后端启动时检查 SQLite 完整性（`PRAGMA integrity_check`）；3) 部署文档中增加定期备份策略 |
| MISS-3 | **并发打卡冲突** — 用户快速多次点击同一任务复选框，虽有限流但有 race condition 可能产生重复 checkin | **中** | 后端打卡接口增加唯一约束检查：`(task_id, checkin_time, user_id)` 复合唯一索引或应用层幂等校验 |
| MISS-4 | **AI 返回的 scheduled_date 超出合理范围** — 如 scheduled_date=2099-01-01 或日期早于今天 | **中** | 后端 /ai/generate-plan 返回后增加日期校验：scheduled_date 须在 [今天, 今天+365天] 范围内，超出的过滤或纠正 |
| MISS-5 | **批量创建子任务中部分失败** — POST /tasks/batch 包含 30 个子任务，第 15 个格式错误导致失败，前 14 个是否需要回滚 | **中** | 明确事务策略：小批量（≤30 个）建议全量事务，任意一个失败则全部回滚并返回错误详情；或支持"部分成功"模式，返回成功列表和失败原因 |

### 3.3 spec.md 边界条件与 PRD 的一致性问题

PRD §10（边界条件与异常处理）覆盖了 17 个场景，spec.md 本身没有独立的边界条件章节。建议在 spec.md 中新增「边界条件与错误码」章节，明确各 API 的错误响应格式。

## 四、技术可行性评审

### 4.1 架构风险（RISK — 3 项）

| # | 风险 | 严重度 | 建议 |
|---|-----|:---:|------|
| RISK-1 | **SQLite 并发瓶颈** — SQLite 单写锁在并发打卡场景下可能成为瓶颈；表中无显式索引可能导致扫描性能退化 | **中** | 短期：Task 26 中已规划"添加索引"但未列具体索引名，补充：`idx_tasks_userid`、`idx_checkins_userid_date`、`idx_checkins_taskid`；中期：日均打卡 > 10w 次考虑迁移 PostgreSQL |
| RISK-2 | **Taro + ECharts 小程序兼容性** — ECharts 小程序版（echarts-for-weixin）与 Taro 框架可能产生组件生命周期冲突 | **中** | 开发前在 Taro 3.x + echarts-for-weixin 环境下做 PoC 验证折线图和环形图渲染；如果兼容性差，备选方案：Canvas 2D API 手绘简易图表 |
| RISK-3 | **DeepSeek API 稳定性和 JSON 格式可靠性** — 大模型存在概率性输出不规范 JSON 的风险，spec.md §4.4 中写了"后端有数据校验"，但未定义校验失败后的处理逻辑 | **高** | 1) Prompt 中强调"只返回合法 JSON，不包含 markdown 代码块标记"；2) 后端增加 OpenAI Function Calling 模式约束输出；3) 增加重试机制（最多 3 次，每次调整 Prompt）；4) 3 次全部失败时返回 500 + "AI 规划暂时不可用，请稍后重试或使用自定义计划" |
